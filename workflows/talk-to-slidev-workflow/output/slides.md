<br>

> 你有没有这样的经历？

# 第一代：Prompt（提示词）

## 原始时代：每次对话都从零开始

```
用户: 帮我写一个函数，要求用 TypeScript，用 async/await，
     错误处理用 try-catch，变量命名用 camelCase...

AI: 好的，这是代码...

--- 新对话 ---

用户: 帮我写另一个函数，要求用 TypeScript，用 async/await，
     错误处理用 try-catch，变量命名用 camelCase...  ← 又要说一遍！
```

**痛点**：重复劳动、容易遗漏、无法积累

---

# 第二代：Custom Instructions（自定义指令）

## 进步：AI 终于能"记住"你的偏好了

```yaml
# ChatGPT / Claude 的 Custom Instructions

关于我：
- 我是一名全栈工程师
- 主要使用 TypeScript 和 React

你应该：
- 代码使用 TypeScript
- 遵循函数式编程风格
- 提供简洁的解释
```

**解决了**：基本偏好的持久化
**新痛点**：只能存一份、无法分场景、不够专业化

---

# 第三代：MCP（Model Context Protocol）

## 突破：AI 能连接外部世界了

```
┌─────────────┐     MCP      ┌─────────────────┐
│   Claude    │ ◄──────────▶ │  外部服务/数据   │
└─────────────┘              └─────────────────┘
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
               数据库查询      文件系统操作      API调用
```

**解决了**：AI 获得了"动手能力"
**新痛点**：它会动手了，但不知道该怎么做得专业

---

# 问题来了：有工具，但没专业知识

想象一下这个场景：

> 你给一个新手程序员一台配置好的电脑（MCP）
> 但没有告诉他团队的代码规范、设计模式、最佳实践
>
> 结果：他能写代码，但写出来的不是你想要的

<br>

**这就是 MCP 之后的核心问题：**

AI 有了工具，但缺少 **专业知识** 和 **工作流程**

---

```
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌──────────────────┐
                    │   Claude Code    │

                    └──────────────────┘
```

**Skills = 让 AI 像专家一样思考和工作**

---

**一个 Skill，就是一本针对特定领域的"专家手册"**

# Skills 解决的三大核心问题

## 1. 知识封装 —— 专业能力可复用

```
之前：每次都要解释"什么是好的 PPT 结构"
现在：ppt-creator Skill 里已经写好金字塔原则、断言式标题等
```

## 2. 流程标准化 —— 质量有保障

```
之前：AI 可能跳过关键步骤
现在：Skill 定义完整的 9 阶段工作流，确保不遗漏
```

## 3. 知识共享 —— 团队/社区协作

```
之前：每个人都在重复造轮子
现在：一人写好 Skill，全团队/全社区受益
```

---

# 什么时候该用 Skills？

## 适合用 Skills 的场景

| 场景 | 例子 |
|------|------|
| 重复性专业任务 | 代码审查、文档生成、数据分析报告 |
| 有固定流程的工作 | PPT 制作、Git 提交、PR Review |
| 需要专业知识的领域 | 系统架构设计、安全审计、性能优化 |
| 团队需要统一标准 | 代码规范、文档模板、测试策略 |

## 不需要 Skills 的场景

- 一次性的简单问答
- 没有固定模式的创意工作
- 需求每次都完全不同

# Skills 的本质：一个 Markdown 文件

```markdown
---
name: commit-message
description: "生成规范的 Git commit 消息。
  触发词：commit、提交信息、git message"
---

# Git Commit Message Generator

## 何时使用
当用户完成代码修改，需要提交时

## 工作流程
1. 分析 git diff 的变更
2. 识别变更类型（feat/fix/docs...）
3. 生成简洁的英文描述

## 格式规范
type(scope): subject

## 示例
- feat(auth): add OAuth2 login support
- fix(api): resolve null pointer exception
```

---


# 案例 1：ppt-creator —— 复杂 Skill 的典范

## 它解决什么问题？

> "做 PPT 很痛苦，AI 生成的又不够专业"

## 它怎么解决的？

```
封装知识：金字塔原则、断言式标题、数据可视化选择
定义流程：9 阶段工作流（输入 → 结构 → 内容 → 设计 → 交付）
质量保障：自动评分，< 75 分自动优化
安全默认：缺信息不卡住，用默认值继续
```

**结果**：一句"帮我做个 PPT"，输出专业级演示文稿

---

# 案例 2：mermaid-diagrams —— 简洁 Skill 的典范

## 它解决什么问题？

> "每次画流程图都要查语法"

## 它怎么解决的？

```markdown
# 不是长篇大论，而是快速参考手册

## Flowchart
flowchart TD
    A[Start] --> B{Valid?}
    B -->|Yes| C[Process]

## Sequence Diagram
sequenceDiagram
    A->>B: Request
    B-->>A: Response
```

**370 行，覆盖 10 种图表类型，即查即用**

---

# 两种 Skill 设计哲学的对比

| | ppt-creator | mermaid-diagrams |
|--|-------------|------------------|
| **复杂度** | 高（400+ 行 + 8 个 references） | 低（单文件 370 行） |
| **适用场景** | 复杂创作任务 | 工具使用参考 |
| **知识类型** | 方法论 + 流程 + 规范 | 语法速查 |
| **交互方式** | 多轮引导 | 即查即用 |
| **设计理念** | 教 AI 像专家一样思考 | 给 AI 一本速查手册 |

<br>

**启示**：Skill 复杂度要匹配任务复杂度，不要过度设计
